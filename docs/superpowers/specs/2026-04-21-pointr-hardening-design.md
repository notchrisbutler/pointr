# Pointr Hardening Design

## Summary

This design fixes the duplicate-player reconnect bug first, then hardens the Cloudflare Worker and Durable Object configuration in a second phase. The key change is replacing name-based reconnect identity with a stable client-generated identifier, backed by tests that cover reconnect, blank-name joins, name collisions, and stale-socket behavior.

## Goals

- Eliminate duplicate logical players during disconnect and reconnect.
- Preserve host status, vote state, and observer state across reconnects.
- Add automated tests for the Durable Object reconnect contract and the narrow client reconnect contract.
- Convert Worker configuration to `wrangler.jsonc` and remove current config parsing warnings.
- Align the Worker and Durable Object setup with current Cloudflare guidance, including consistent route validation and a real TypeScript verification path.

## Non-Goals

- Redesign the UI or the session feature set.
- Introduce a full browser E2E framework in the first pass.
- Perform unrelated refactors outside the session identity, test, and platform-hardening scope.

## Current Problems

### Reconnect identity bug

The Durable Object currently identifies reconnects by display name. This fails when the server canonicalizes the name differently from what the client later resends, especially for blank names that become random emoji names and for colliding names that receive suffixes. The client also tries to determine its own player entry by scanning the player list by name, which is incorrect for blank names and fragile for duplicate names.

### Missing test coverage

The repository currently has no automated tests covering the Durable Object or reconnect behavior. That leaves session identity and transport edge cases unprotected.

### Cloudflare configuration drift

The checked-in Worker config currently produces a Wrangler warning, uses TOML rather than the requested JSONC format, and still defines the Durable Object through legacy `new_classes` configuration. Session ID validation is inconsistent across routes, and plain `npx tsc --noEmit` is not currently a usable verification step.

## Phase 1: Reconnect Identity And Tests

### Identity model

Reconnect identity will change from display name to a stable `clientId` generated by the client and persisted locally. A player's display name becomes presentation state, not identity.

Each player connection record will carry at least:

- `clientId`
- canonical display name
- current vote
- observer flag
- host flag

The Durable Object attachment for each accepted WebSocket will store the same identity and session fields needed to rebuild in-memory state after hibernation.

### Join and reconnect flow

1. The session page loads and creates or restores a stable `clientId`.
2. The client opens the session WebSocket.
3. On open, the client sends a join payload containing `clientId`, requested name, and observer status.
4. The Durable Object resolves the join:
   - if another socket already has the same `clientId`, treat it as a reconnect, preserve session state, and replace the older socket
   - if the `clientId` is new, create a new logical player
5. The Durable Object computes the canonical display name:
   - blank requested names get a random emoji name only once for that `clientId`
   - reconnects keep the existing canonical display name for that `clientId`
   - collisions among different `clientId`s are resolved with suffixes for display only
6. The Durable Object sends a direct `joined` acknowledgement to the connecting socket containing the resolved `clientId`, canonical display name, and current role flags, then continues broadcasting the shared session state.

### Client reconnect behavior

The client reconnect loop will be generation-aware so only the latest socket instance can schedule another reconnect. This prevents server-initiated replacement of an older socket from creating a second follow-up reconnect attempt from stale client state.

The client will stop determining `amHost` or its canonical name by scanning the player list using the raw local name. Instead, it will use the authoritative data from the direct `joined` acknowledgement and subsequent state updates.

### Phase 1 test design

Phase 1 introduces a small Worker-compatible automated test setup. The design does not require a full browser E2E harness. The tests should be fast, deterministic, and focused on the Durable Object session contract.

The Durable Object test block will cover:

- join with explicit name creates one player
- reconnect with same `clientId` keeps one logical player
- reconnect preserves vote, observer state, and host state
- blank-name join assigns one canonical emoji name and reconnect keeps it
- two distinct `clientId`s requesting the same name get different display names
- replacing an old socket does not leave duplicate players after close and error paths

The client-focused narrow tests will cover:

- `clientId` creation and persistence
- reuse of the same `clientId` across reconnect attempts
- stale socket close events do not trigger extra reconnect attempts
- client self-identification no longer depends on name matching

### Phase 1 verification

Phase 1 is complete only when all of the following are true:

- automated reconnect tests pass
- a local runtime repro no longer shows duplicate players for blank-name reconnects
- manual verification confirms host transfer, observer mode, and vote persistence still work across reconnects

## Phase 2: Worker And Platform Hardening

### Worker configuration

Replace `wrangler.toml` with `wrangler.jsonc`.

The new JSONC config will:

- preserve the current bindings and rate limits
- express routing in valid current Wrangler syntax
- eliminate the current parsing warning
- express Durable Object migrations in current JSONC format

### Durable Object storage configuration

The design target is a SQLite-backed Durable Object configuration rather than the legacy class definition. Because changing Durable Object backend configuration can affect persisted state, the implementation plan must treat this as a deliberate migration decision.

The implementation plan must explicitly choose one of these rollout strategies:

- acceptable reset of existing session state because sessions are ephemeral
- controlled namespace/class migration with documented user impact

The plan must not silently switch the backend without describing the operational effect.

### Route validation

Session ID validation will be centralized and applied consistently to:

- session page route
- session info route
- session WebSocket route

Invalid IDs should fail before routing to the Durable Object namespace.

### TypeScript verification

The repository will be updated so `npx tsc --noEmit` becomes a real verification command. That includes the Worker environment types, the necessary Web platform libs, and inclusion or generation of Cloudflare binding types in a way that matches the repo's actual build flow.

### Rate limiting review

The current IP-based keys should be revisited. For this app, the design goal is to reduce avoidable false positives while keeping the implementation simple. The implementation plan should evaluate route-specific keys that incorporate stable resource identifiers where practical, while acknowledging that pre-join requests may still need some coarse fallback.

### Phase 2 verification

Phase 2 is complete only when all of the following are true:

- `wrangler` no longer warns about configuration parsing
- `npx wrangler deploy --dry-run` succeeds with the new config
- `npx tsc --noEmit` succeeds
- automated tests from Phase 1 still pass
- invalid session IDs are rejected consistently before Durable Object routing

## Risks And Constraints

- The reconnect fix changes the client-server identity contract and must be rolled out consistently on both sides.
- Durable Object backend or migration changes can affect existing persisted state and must be treated explicitly.
- Hibernation-safe behavior must be preserved; any new per-connection fields needed after wake-up must be present in the serialized attachment.

## Recommended Implementation Order

1. Add Worker-compatible test scaffolding.
2. Add failing reconnect and client identity tests.
3. Implement `clientId`-based reconnect identity.
4. Verify the local blank-name reconnect repro is fixed.
5. Convert `wrangler.toml` to `wrangler.jsonc`.
6. Apply route validation and TypeScript environment fixes.
7. Make the Durable Object storage migration decision and implement it.
8. Re-run full verification.

## Open Decisions Resolved In This Design

- Work will be split into two phases rather than one combined refactor.
- The Worker config target format is `wrangler.jsonc`.
- Tests are required as part of the reconnect fix, not as follow-up work.
